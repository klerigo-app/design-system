# Design Sync Notes

## Known warnings (triaged)

- `[FONT_MISSING] "Trebuchet MS"` — this is a fallback in the `--font-display` stack (`'Baloo 2', 'Trebuchet MS', sans-serif'`), not the intended font. Baloo 2 ships its own `@font-face` via `@fontsource/baloo-2` and is what actually renders. Trebuchet MS is a Windows system-font fallback for the rare case Baloo 2 fails to load — no action needed.

## Config decisions

- No `cfg.cssEntry` set. The package ships CSS as two separate dist files (`dist/index.css` = Tailwind/component styles, `dist/fonts.css` = `@fontsource` imports + `--font-*` custom properties) that are NOT concatenated by the build (`package.json`'s `build` script just copies `fonts.css` into `dist/` — it's a separate export, `./fonts.css`). Storybook's `preview.tsx` imports both, so relying on `[CSS_FROM_STORYBOOK]` (scraping the compiled iframe CSS) captures the full closure correctly, whereas setting `cssEntry` to just `dist/index.css` caused `[TOKENS_MISSING]` for `--font-mono`/`--font-body`/`--font-display`. Leave `cssEntry` unset.
- `titleMap: {"Overview": null}` — `Tokens/Overview` is a token-showcase story (color ramps, radii), not a real component export; excluded.
- `overrides.Modal: {cardMode: "single", primaryStory: "TypeToConfirmDelete"}` — Modal renders via a fixed-position overlay/portal (`[GRID_OVERFLOW]`/`[RENDER_THIN]` in validate), so no grid layout can present its stories side-by-side. `TypeToConfirmDelete` was picked as the primary story since it demonstrates the type-to-confirm gating that differentiates this Modal from a generic one.
- `overrides.Toast: {skip: ["feedback-toast--interactive"]}` — Toast's `Interactive` story is an interaction-only playground (a `ToastProvider` + buttons that fire toasts on click); statically it shows only the button bar, and the compiled preview rendered it blank. Root cause of the blank + the product-card `useToast must be used within a <ToastProvider>` error: when the story MODULE is recompiled for the preview, its relative `./ToastProvider` and `./useToast` imports resolve to different `ToastContext` instances, so the hook throws even though wrapped. This is a preview-compilation artifact, NOT a bundle defect — the shipped bundle's `ToastProvider`/`useToast` share one context and work (verified by the package's own 178 vitest tests). The 4 static stories (Info/Success/Warning/Error) render `<Toast>` directly (no provider, no portal) and all grade `match`; skipping Interactive removes the blank cell and clears the product-card error. Do NOT `cardMode: single` here — the 4 static stories make a good multi-variant grid card on their own.
- `readmeHeader: ".design-sync/conventions.md"` — authored conventions covering the Tailwind-preset styling idiom (real class names like `bg-coral-500`, `shadow-lift-coral`, `rounded-card`), no-provider-needed setup, and an idiomatic composition example. Validated every named class/prop against the compiled `_ds_bundle.css` and component source before shipping.

## Re-sync risks

- **Toast (added 2026-07-09)**: overlay/portal component like Modal. Grade only the 4 static stories (Info/Success/Warning/Error) — `Interactive` is skipped (see Config decisions). Toast's generated `Toast.prompt.md` documents ONLY the presentational `<Toast>` (controlled `variant`/`title`/`body`/`links`/`duration`/`onDismiss`); the imperative `ToastProvider` + `useToast()` system is bundle-exported but not in the prompt (its only story is the skipped Interactive one). If a future story demonstrates the provider statically, un-skip it and grade. The `duration` prop is `number | null` in source but the extracted `.d.ts` narrows to `number` (JSDoc still notes `null` persists) — cosmetic dts-extraction quirk, not worth chasing.
- **Button / Modal changed (2026-07-09)**: both showed as `changed` vs the anchor (outline Button variant + Modal `size` prop landed after the 3rd sync) and re-graded all-`match`. Modal's gated-reference caveat below still applies. If a future sync shows them `changed` again with no source edit, suspect a sourceKey/toolchain shift.
- **Table (added 2026-07-08)**: both stories (`Default`, `Narrow Container Scrolls`) graded `match` on first capture — no owned preview needed. `minWidth` prop drives the horizontal-scroll behavior shown in `NarrowContainerScrolls`; verified visually in the compare sheet.
- **Modal's compare sheets always show a gated storybook reference.** The storybook side of every Modal story's compare screenshot shows only the "Abrir modal" trigger button on a flat backdrop — never the open dialog — because storybook's capture crops to the story root element's bounding box, while Modal renders its dialog via `createPortal` to `document.body` with `position: fixed`, painting outside that crop. This is independent of `cardMode` (which only fixes the PRODUCT card grid, not this compare capture). Grade Modal by judging the preview's own render against `Modal.tsx`'s actual logic (variant→icon/badge/button color, `onCancel` guard, confirmation-value gating), not by comparing to the reference image — a future re-grade must not mistake this permanent gating for a regression.
- All 24 components graded `match` on every story on the first sync (2026-07-06) — no owned previews were needed anywhere; the generated story-compiled previews were faithful out of the box. If a future re-sync shows unexpected `mismatch`es on components that were clean here, suspect a build/toolchain change (Vite, esbuild, Tailwind version bump) rather than a story-source issue.
- **Second sync (2026-07-07)**: the EspañoLenka→Klerigo brand rename changed every color/token name, so `styleSha` changed even though only `Card` had an actual source edit (non-interactive by default). This produced `render_churn` across all 23 otherwise-unchanged components — expected, since a token rename repaints everything. The driver's canary spot-check (Modal, Radio, Heading, Chip, SegmentedControl) confirmed all five render correctly with the new Klerigo palette/branding, so grades carried forward with no re-grade needed. If a future sync shows the same `render_churn` trigger after a purely cosmetic/token-level change, the canary-confirm path (not a full re-grade) is the right response.
